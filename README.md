# Git Commit Message Convention

깃 커밋 메시지를 **일관성 있고 명확하게** 작성하기 위한 컨벤션입니다.  
팀 협업이나 오픈소스 프로젝트에서 다른 사람이 변경 사항을 쉽게 이해할 수 있도록 작성합니다.

---

## 1. 기본 구조

- **타입(Type)**: 커밋의 목적을 나타내는 키워드  
- **범위(Scope)**: 변경된 모듈, 기능, 파일 등 (선택)  
- **요약(Summary)**: 최대 50자 이내로 간결하게 작성  
- **본문(Body)**: 왜 변경했는지, 이전 코드 대비 어떤 점이 달라졌는지 (선택)  
- **꼬리말(Footer)**: 관련 이슈 번호나 참고 사항 (선택)

---

## 2. 추천 타입
| 타입 | 의미 | 예시 |
|------|------|------|
| feat | 새로운 기능 추가 | feat(auth): 애플 로그인 기능 추가 |
| fix | 버그 수정 | fix(network): 데이터 fetch 시 크래시 수정 |
| docs | 문서 관련 수정 | docs: README.md 오타 수정 |
| style | 코드 스타일 변경 (기능 변경 없음) | style: indent, 공백, 세미콜론 수정 |
| refactor | 코드 리팩토링 | refactor: MemoViewModel 구조 개선 |
| perf | 성능 개선 | perf: 이미지 로딩 속도 최적화 |
| test | 테스트 코드 추가/수정 | test: MemoListView 단위 테스트 추가 |
| chore | 기타 변경 (빌드, 패키지 업데이트 등) | chore: 패키지 버전 업데이트 |

## 4. 작성 팁
1. **한 커밋에는 한 가지 변경만** 포함시키기  
2. **과거형 대신 명령형**으로 작성 (`Fix bug` X → `fix bug` O)  
3. **짧고 명확하게 요약**하고 필요 시 본문에 상세 설명 추가  
4. **관련 이슈가 있으면 꼬리말에 연결** (`Closes #12`)  

---

> 💡 커밋 메시지를 일관성 있게 작성하면, 나중에 히스토리 관리와 코드 리뷰가 훨씬 쉬워집니다.
